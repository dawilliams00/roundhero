## What this is

RoundHero is a D&D 5e character tracker web app: Flask/SQLAlchemy backend, React frontend, Postgres on Render in production (SQLite locally). Single user currently (the owner, testing with their own character "Syric Nightbloom," a level 13 Shadar-kai Wizard imported from a D&D Beyond PDF). No test suite exists yet — there is no CI, no test runner setup beyond the default `react-scripts test` stub, and no backend test framework. ## Architecture

### Two parallel character-build paths feed one shared shape

Every character ends up with the same three JSON blobs — `tracker_data`, `spell_data`, `ae_data` — but they're produced by two completely different code paths that must be kept in sync:

- **Manual creation** (`engine/character_engine.py`): builds from `engine/content_packs.py`, a hand-authored dict of all 12 classes (hit die, saves, per-level features, spell slot progression). `build_tracker_data()` / `build_spell_data()` / `build_ae_data()`.
- **PDF import** (`engine/pdf_import.py`): `parse_character_pdf()` reads a D&D Beyond character sheet PDF. Critically, the visible D&D Beyond PDF is mostly a printed background — **the actual character data lives in PDF form-field widgets** (read via PyMuPDF/`fitz`, not the text layer; `pdfplumber`/text extraction will only return field labels). Field names follow D&D Beyond's fixed layout (`CLASS  LEVEL` — two spaces, `Wpn Name`/`Wpn Name {N}` inconsistently numbered, `spellName{N}`/`spellHeader{level}` for the spell table, etc.) — see the regexes in `pdf_import.py` for the exact contract. This path produces `build_ae_data_from_features()` instead of `build_ae_data()`.

Both paths must assign **`cost_type`** on every `ae_data` entry to one of `action` / `bonus_action` / `reaction` / `cast_spell` / `free_action` / `passive` — this is what the frontend's turn-economy bucket-tracking keys off. **The section an ability is filed under IS the turn-economy bucket** for action/bonus_action/reaction/cast_spell types (`ActionEconomyTab.js`'s `bucketForAbility(ability, section)` — `haste_action` is the one cost_type that maps to a different bucket name, `'Haste'`). A feature can display correctly under the right section header while still being inert for bucket-tracking if `cost_type` doesn't match the section it's filed under — this exact bug existed for months because both builders hardcoded `cost_type: "feature"` regardless of section, and a near-identical version of it existed in `CustomAbilityModal.js` (an independent `cost_type` dropdown that didn't have to agree with the chosen section) until that was fixed to *derive* cost_type from section automatically.

`"Cast a Spell"` appears as its own stock entry in **Action, Bonus Action, and Reaction** sections (`STOCK_ACTIONS`/`STOCK_BONUS`/`STOCK_REACTIONS` in both builders) — clicking CAST in a given section opens `CastSpellPickerModal` filtered to spells whose `casting_time` matches that bucket (`spellCastBucket()` in `utils/dnd.js`), so Counterspell won't show up under Action and Fireball won't show up under Reaction. Every ability — even ones with no tracked charge (Disengage, Dodge, etc.) — gets a USE button that marks the turn-bucket consumed even with nothing to decrement.

### Re-sync: PDF-imported characters can absorb engine improvements without losing live state

`Character.source_pdf` stores the raw uploaded PDF bytes. `POST /api/characters/<id>/resync` re-runs `parse_character_pdf()` on those stored bytes and merges the result into the existing character via `resync_character()` in `pdf_import.py`. The merge logic distinguishes parser-owned data from player-added data using a `"_source": "pdf"` tag on features/spells/items: anything tagged `pdf` is replaced by the fresh parse (so parser bug fixes propagate), anything untagged (added later through the UI — custom abilities, custom spells, custom items) is preserved untouched. Live/mutable state (HP, feature/slot/charge `current` counts, manual AC/initiative overrides) is also preserved across a resync, clamped to the new `max` values. When changing `pdf_import.py`'s parsing logic, the fix only reaches already-imported characters when the user clicks "Re-sync from PDF" — it does not retroactively touch stored `ae_data`/`tracker_data` on its own. The button lives on the **Notes tab** (`NotesTab.js`), labeled "↻ Re-sync from PDF".

Inventory items have the same staleness problem but no resync mechanism: `ItemBrowserModal`/`ItemDetailModal` copy a snapshot from `magic_items.json` into the character's `tracker_data.inventory.items` at add-time, with no link back afterward. A later fix to the reference data (wrong description, missing buff, etc.) never reaches characters who already own the item. `InventoryTab.js`'s `refreshItem()` (wired to a 🔄 **Refresh** button on `ItemDetailModal`) re-pulls static fields (description/weight/rarity/buffs/granted_spells/charge mechanics) from `/api/content/items` by name match and merges them in, while leaving quantity/equipped/attuned/current-charges untouched. This is an explicit per-item action, not automatic — it was deliberately kept manual rather than auto-merging on every character load, since auto-merging description/rarity could silently clobber a player's intentional edits to a custom-tweaked item.

### `tracker_data` is a single free-form JSON blob, not normalized columns

`models/character.py` stores `tracker_data`, `spell_data`, `ae_data`, `notes`, `ability_scores` each as a `Text` column holding JSON (property getters/setters handle the (de)serialization). This means **adding a new field to the character's runtime state never requires a migration** — components just read/write nested keys via `saveTrackerData()`. The known shape of `tracker_data` (no schema enforces this, components rely on convention):
```
features: { [name]: {current, max, rest_type, action, description, _source?, spell_picker?, tucked_spell?, tucked_level?} }
spell_slots: { [level]: {current, max} }
item_charges: {}            # legacy, mostly superseded by inventory.items[].charges
conditions: [string]         # checkable via the header's Conditions button (ConditionsModal.js); excludes "Exhaustion", which is its own field below
exhaustion: 0-6              # plain integer, stepped via the header's PMStat - no mechanical effects enforced, same philosophy as conditions
settings: {ruleset?: '2014'|'2024', exhaustion_rules?: {mode:'raw'|'homebrew', name?, description?}}  # set via the gear-icon SettingsModal.js
hp: {current, max, temp, max_override}
hit_dice: {current, total, die_size}
ac, initiative, inspiration, in_initiative: scalars
traits: {resistances, immunities, vulnerabilities, advantages, disadvantages: [string | {name, description}]}
active_effects: [string]    # e.g. "Hasted" — addable/removable from both TrackerTab.js and the header (EffectAdder in CharacterHeader.js)
inventory: {currency: {cp,sp,ep,gp,pp}, items: [{name, quantity, weight, rarity, equipped, attunement, attuned, charges?, granted_spells?, buffs?, cost_type?, _source?, is_weapon?, weapon_category?, weapon_range?, damage_dice?, damage_type?, properties?, range?, two_handed_damage?, proficient?, two_handed?}]}
save_proficiencies, skill_proficiencies: [string]
active_creatures: [{id, creature_name, instance_name, hp:{current,max}}]  # summoned creatures, see Bestiary below
```
`items[].cost_type` (`action`/`bonus_action`/`reaction`/`free_action`, or unset) is player-assigned per item via a dropdown in the Action Economy tab's ITEMS section, since the 1,267-item reference DB has no notion of "what action does activating this cost" — using a charge while `in_initiative` marks that bucket consumed the same way a feature would. `features[name].spell_picker` flags a "tuck & release" ability (see below).

`Character.source_pdf` is the one real column that *did* require a migration. Since `db.create_all()` only creates missing tables and never alters existing ones, `app.py` has a `PENDING_COLUMNS` list + `_apply_pending_migrations()` that runs additive `ALTER TABLE ... ADD COLUMN` statements on every startup (Postgres `IF NOT EXISTS`, SQLite best-effort). Any future new *column* (not a `tracker_data` key) must be added to that list or it will silently not exist in the deployed Postgres DB. A brand-new *table* (like `CustomContent`, below) doesn't need an entry here — `db.create_all()` creates missing tables fine, it just never alters existing ones.

### Shared content library: `CustomContent` table, not per-character JSON

Player- or DM-created homebrew content is meant to become available to *every* character, not just the one it was typed in on ("a player or DM creates a feat once, it's in the DB forever, searchable by anyone" — the owner's explicit framing). `models/custom_content.py`'s `CustomContent` is a generic table: `user_id`, `content_type` (`"spell"` | `"feat"`, more types later), `name`, and a free-form JSON `data` column — deliberately generic so a future content type doesn't need its own table. `routes/content.py` has GET/POST pairs (`/content/spells`, `/content/feats`) where GET merges the static JSON-file content with all `CustomContent` rows of that type (**unfiltered by user** — homebrew is shared across everyone, not scoped to its creator) and POST creates a new row. `CustomSpellModal.js` and `CustomAbilityModal.js` both POST here in addition to attaching the new spell/feature to the character that created it. `FeatBrowserModal.js` is the read side for feats — lets any character search the library and add an existing entry instead of retyping it (`TrackerTab.js`'s "Browse Feats" button — moved there from the Action Economy tab along with "+ Custom", since adding/browsing feats is conceptually tracker-management, not a turn-economy action); `SpellBrowserModal.js` already did the equivalent for spells via the existing `/content/spells` GET.

Items use this same table but split into **two** `content_type`s instead of one, because "fix the canon entry for everyone" and "make my own independent variant" are both real, distinct asks for items specifically (a magic item's printed stats sometimes just need correcting, the way a spell or feat rarely does once it's in the static JSON). `item_override` rows are matched **by name** against the canon list in `magic_items.json` — `GET /content/items` replaces a canon entry with its matching override's data wholesale (tagging it `_source: "canon_override"`), and the static file on disk is never touched. Plain `item` rows are fully independent duplicates/new homebrew, same as monsters. `ItemBrowserModal.js`'s detail view is where this surfaces: canon items get both "✏️ Admin Edit" (upserts the override via `PUT /content/items/override`) and "📋 Duplicate" (`POST /content/items`, named "X (Homebrew)" by default); an already-overridden canon item additionally gets "↺ Revert" (`DELETE /content/items/override/<name>`); an independent homebrew item gets "✏️ Edit"/"🗑️ Delete" instead (`PUT`/`DELETE /content/items/<custom_id>`). An existing character's already-owned copy of an admin-edited item picks up the fix the same way any other canon data fix does — the per-character 🔄 Refresh button in `InventoryTab.js`, unchanged. Custom feats got the same `PUT`/`DELETE /content/feats/<custom_id>` treatment as monsters (edit/delete via `FeatBrowserModal.js`'s "✏️ Edit"/"🗑️" buttons on `_source === 'custom'` rows, reusing `CustomAbilityModal.js` with an `editingFeat` prop that skips the "attach to my character" step).

Campaign and encounter screens reuse `FeedbackModal` with a context label, so reports from those pages include the campaign/encounter name in the email subject/body. This depends on the existing Render feedback SMTP env vars; no separate campaign feedback backend exists.

A third variation on the same table: `ruleset` rows (currently just homebrew exhaustion-rules name+description) are upserted **by name** via `PUT /content/rulesets` rather than created fresh each time — `SettingsModal.js` calls this automatically on every blur of the homebrew name/description fields, so one player typing out a homebrew exhaustion table is automatically searchable by everyone else without a separate "publish" step. `RulesetBrowserModal.js` is the search/import side, opened from a "🔍 Browse Homebrew Rulesets" button next to the homebrew fields; importing just copies `{name, description}` into the current character's `settings.exhaustion_rules`, same shallow copy semantics as adding a feat from the library. `ruleset_type` (currently always `"exhaustion"`) is stored in `data` and filtered via a query param so other ruleset categories can reuse `/content/rulesets` later without a new endpoint.

### Generic "tuck & release a spell" ability (ported from the Pi tracker's Cartomancer feat)

A feature with `spell_picker: true` in `tracker_data.features` renders specially in `ActionEconomyTab.js`: instead of a plain USE button it gets an OPEN button that launches `SpellTuckModal.js`. That modal lets the player pick **any spell from the full class spell list** (fetched live from `/api/content/spells?class_name=X`, not limited to the character's own known spells — this is the actual homebrew rule and was a real bug before the fix) to "tuck," then later "cast" the tucked spell for free (decrements the feature's `current` count, no slot spent). `CustomAbilityModal.js` has a checkbox to flag a new custom ability this way, so this is a reusable capability, not a one-off hardcoded "Cartomancer" feature. `class_name` for PDF-imported characters is often a raw string like `"Wizard 13"` or multiclass `"Wizard 10 / Fighter 3"` — `parseClassLevels()` in `utils/dnd.js` is the shared parser for splitting that into `{className, level}` parts; the spell-list fetch currently only uses the *first* class found.

### Feat-granted known spells with a free-use charge and an optional ability override

A different mechanic from "tuck & release" above, for feats like Draconic Healing (Gift of the Metallic Dragon) that grant a spell *permanently known*, castable a limited number of times without a slot (e.g. once per long rest), *and* normally via any real slot the character has — with the spell's attack/DC fixed to a specific ability chosen when the feat was taken, independent of (and instead of) whatever the character's class would otherwise use. `CustomAbilityModal.js`'s three "what kind of ability is this" checkboxes (spell-like action / tuck & release / grants-a-known-spell) are mutually exclusive — checking one clears the other two via `setExclusiveFlag()`. Checking "grants a known spell" reveals a spell-search autocomplete (same pattern as `AddItemModal.js`'s granted-spell picker) and an ability dropdown (blank = class default).

On submit, this writes to **two** places at once: `tracker_data.features[name]` gets `granted_spell`/`ability_override` keys alongside the usual `current`/`max`/`rest_type` (the "Uses" field is the free-cast count), and the resolved spell object is pushed into `spell_data.known_spells` tagged `{granted_by, ability_override, free_use_feature}` — `free_use_feature` is what lets `SpellDetailModal.js` find the right feature's charge later. `TrackerTab.js`'s `addFeatFromLibrary()` does the same two-place write when importing this kind of feat from the shared library (it re-fetches `/content/spells` to resolve the spell by name, since the library payload only carries the name).

`SpellDetailModal.js` reads both tags off the spell object directly, no feature cross-reference needed for display: `spell.ability_override` swaps `getSpellcastingBlocks()`'s class-based blocks for a single `getAbilityOverrideBlock()` result (same attack/DC math, just keyed off one fixed ability — this is also what makes the spell show *any* attack/DC block at all for a character with no spellcasting class). `spell.free_use_feature` looks up that feature's `current` count; if it's > 0, a "Cast Free" button appears via `doFreeCast()` (calls the existing `useFeature()`, always at the spell's own base level, no upcasting) — shown *alongside* the normal slot-cast button when slots exist, or *instead of* the "No spell slots available" message when they don't. This second part is the actual fix for "a non-caster with a feat-granted spell couldn't cast it" — `SpellsTab.js`'s `castable` check and `hasAvailableSlot()` only ever looked at spell slots, so a character with zero slots had the Cast button disabled in the list before the modal even opened; `hasFreeUse()` was added alongside `hasAvailableSlot()` specifically to fix that.

**Pre-existing characters who already had an ad hoc "granted spell" set up before this existed** (e.g. by adding the spell manually via Spells tab's "+ Custom Spell" with just a `granted_by` display string, no `free_use_feature`/`ability_override`) won't automatically get the new behavior — those tags have to actually be present on the spell object. Fixing an existing character means removing the old ad hoc entry and recreating the feat through `CustomAbilityModal.js`'s new checkbox so the tags get written correctly; there's no migration that retrofits old data.

### Spell attack modifier / save DC and damage rolling

`getSpellcastingBlocks(classNameRaw, abilityScores, totalLevel, items)` in `utils/dnd.js` returns one `{className, ability, attackMod, saveDC}` block per spellcasting class found via `parseClassLevels` + a hardcoded `SPELLCASTING_ABILITY` map — multiclass characters with two casting classes get two blocks. It folds in `itemSpellBonuses(items)`, which sums `spell_attack_modifier`/`spell_dc_modifier` buffs from items that are **both `equipped` and, if `attunement` is required, `attuned`** (Staff of the Magi/Robe of the Archmagi only grant their bonus while held/attuned per RAW — gating on `equipped` alone was wrong and got fixed). These blocks render in the header (under HP), the Spells tab header, and inside `SpellDetailModal`.

Spell damage uses **structured fields already present in `spells.json`** (`damage_dice`, `damage_type`, `save_type_abbr`, `is_attack`, `attack_type`, `higher_level`) rather than parsing the description text — much more reliable than it sounds, the data was already there. `scaleSpellDamage(spell, castLevel)` scales the dice up via a regex against `higher_level`'s "increases by NdM for each slot level above Xth" phrasing; it only understands that one common phrasing, so spells that scale a different way (e.g. Magic Missile's extra-missile wording) aren't scaled — known, accepted limitation. Casting a damage spell shows a **"Roll Damage?" button** (not an auto-roll — the owner explicitly wants to choose when to roll) with a Reroll/Done flow afterward, instead of the popup just confirming the cast and closing.

One subtlety: when a spell is resolved from an item's fixed `granted.cast_level` (e.g. Fireball always cast at 7th via Staff of the Magi's charges, in `ItemSpellsModal.js`'s `resolveSpell()`), `level_int` gets overwritten to that cast level for display — but `scaleSpellDamage` needs the spell's *true* base level to know how many levels above base it's being upcast. `resolveSpell()` preserves that under `base_level_int`; `scaleSpellDamage` checks `spell.base_level_int ?? spell.level_int`. Forgetting this field is exactly the bug that shipped once (Fireball-via-staff showed base 8d6 instead of scaled 12d6) and got fixed.

### Generalized item stat buffs (AC, ability scores, saves) — and three bugs that made `buffs` mostly inert

`computeItemBonuses(items)` in `utils/dnd.js` is the one place that walks a character's `inventory.items` and aggregates every item's `buffs` array, gated the same way `itemSpellBonuses` always was: item must be `equipped`, and `attuned` too if `attunement` is required. Two buff shapes exist: ADD-mode (`ac_base`, `saving_throw_modifier`, `spell_attack_modifier`, `spell_dc_modifier` — these sum across items) and SET-mode on an ability key (`{stat:"STR", mode:"set", value:19}` for things like Headband of Intellect/Belt of Giant Strength — RAW these don't stack and don't apply if your score is already higher, so `effectiveAbilityScores(abilityScores, items)` takes the *max* of the raw score and any equipped set-buffs, never lowers it). `CharacterHeader.js`'s AC stat box displays `baseAc + itemBonuses.ac_base` as one merged total (matching how a real character sheet has a single AC number) — the base (manual armor+dex entry) is what actually gets stored; typing a new number into the AC box back-solves the base by subtracting the current item bonus, so re-equipping gear doesn't silently double-count. `calcSaves`/`calcSkills`/`maxPreparedSpells`/`getSpellcastingBlocks` all take an optional trailing `items` param and run ability scores through `effectiveAbilityScores` (plus `saving_throw_modifier` for saves) when it's passed — every call site now passes it except where no inventory exists.

This only works because of a non-obvious fix: **`buffs` previously never made it onto a character's actual inventory item through any of the three ways an item gets there.** `ItemBrowserModal.js`'s "Add from DB" copied every field except `buffs`. `pdf_import.py`'s canonical cross-reference (matches a parsed PDF item against `magic_items.json` by name to backfill `charges`/`granted_spells`/`description`) did the same omission, so PDF-imported and resynced items never got buffs either. `AddItemModal.js`'s edit-save `submit()` built its output object field-by-field and didn't list `buffs` at all, so clicking "Edit" on *any* item silently wiped its buffs even if they'd gotten there some other way. The only path that ever worked was `InventoryTab.js`'s manual 🔄 Refresh button, which explicitly copies `buffs` from the DB — which is almost certainly how Staff of the Magi/Robe of the Archmagi ended up working before this fix. All three gaps are now closed; a PDF "Re-sync" will backfill buffs onto already-imported items going forward.

`magic_items.json`'s ability-score-setting wearables (Headband of Intellect, Gauntlets of Ogre Power, Amulet of Health, Gloves of Soul Catching, the six Belt of Giant Strength tiers, Stormgirdle (Dormant), Titanstone Knuckles (Dormant), Thunderous Greatclub) had empty `buffs: []` in the generated data and were hand-populated with SET-mode entries as part of this fix. Out of scope, deliberately not touched: Manuals/Tomes of ability-score-increase and Potions of Giant Strength are a different mechanic (permanent-on-use / temporary-on-drink, not "while equipped") and still have empty buffs; and the ~90 items whose description text mentions "AC" without a structured `ac_base` buff are an existing data-completeness gap in the one-time 5etools conversion (see the `magic_items.json` note further down) — fixing those would mean auditing each one by hand, not something attempted here. (`weapon_attack_modifier`/`weapon_damage_modifier` *are* now consumed — see Weapons below.)

### `AddItemModal.js`'s `submit()` is a recurring footgun — confirmed twice now

The buffs-dropping bug (above) and a second, separately-discovered bug are the same root cause: `submit()` rebuilds an item's `charges` object from only the three fields the form has inputs for (`current`/`max`/`recharge`), so `recharge_amount` (e.g. Staff of the Magi's `"4d6 + 2"` partial recharge) silently vanished on every edit, and `_recharge_item` in `rest_engine.py` falls back to a *full* recharge whenever `recharge_amount` is missing — this is why the Staff kept fully recharging on long rest even after the buffs fix. Fixed by adding a "Recharge Amount" input and explicitly carrying it through in `submit()`. **Any field this modal's form doesn't have an explicit input for will silently disappear from an item the moment someone clicks Edit → Save** — this has now bitten `buffs` and `charges.recharge_amount`; treat it as a standing hazard, not a one-off, when adding any new item field anywhere in the app.

### Weapons: attack/damage rolling, sourced from `equipment.json` not `magic_items.json`

Unlike magic items, mundane weapons didn't need a data-backfill pass — `backend/data/equipment.json` already had 37 SRD weapons fully structured (`weapon_category` Simple/Martial, `weapon_range` Melee/Ranged, `damage.damage_dice`/`damage_type`, `properties` array, `two_handed_damage` on Versatile weapons), already served via `GET /api/content/equipment`, just never surfaced anywhere in the frontend. `ItemBrowserModal.js` ("Add Item" in `InventoryTab.js`) fetches *both* `/content/items` and `/content/equipment` (filtered to `equipment_category === 'Weapon'`) and merges them into one combined, alphabetically-sorted, searchable list tagged with `_kind: 'item' | 'weapon'` — there's no separate "Add Weapon" button or modal (there was originally; it got folded in here, since to a player browsing for gear a longsword and a potion are both just "items"). Weapon entries map onto an inventory item with `is_weapon: true` plus the weapon fields and never disable "Add" for an already-owned name (owning two daggers/shortswords for two-weapon fighting is normal, unlike magic items). `AddItemModal.js` also grew a full weapon-fields section (category/range/damage/properties/proficient) for homebrew weapons — and critically, every new field is listed explicitly in `submit()`'s output object, because this file's `submit()` is a whitelist rebuild, not a spread, and that's exactly the bug class that silently dropped `buffs` on every edit before (see above) — any future field added to this modal must be added to both the form state *and* `submit()`'s `out`, or it'll quietly vanish the next time someone edits the item.

Weapon attack/damage modifiers (`weapon_attack_modifier`/`weapon_damage_modifier`) are deliberately **not** routed through `computeItemBonuses`'s character-wide pool — `weaponItemBonus(weapon)` in `utils/dnd.js` reads only that one weapon's own `buffs`, because a +1 longsword's bonus must never leak onto a character's other weapons the way AC/save bonuses correctly pool across all equipped items. Both `computeItemBonuses` and `weaponItemBonus` share the same equipped/attuned gating via `isItemActive(item)`, factored out specifically so the two consumers can't drift apart the way the spell-bonus gating once had a real bug (see above). `weaponAbilityMod(weapon, effAb)` picks STR vs. DEX (Finesse takes whichever is better; Ranged uses DEX; everything else — including Thrown melee weapons — uses STR, since `weapon_range` on those is still `"Melee"` in the data).

Rolling happens in `WeaponAttackModal.js` (opened from a new WEAPONS section in `ActionEconomyTab.js`, placed above the existing ITEMS section), modeled on `SpellDetailModal.js`'s roll/Reroll/Done pattern but with attack and damage as two independent rolls (the player can roll just damage without ever rolling an attack through the app, useful if they rolled a physical d20). `proficient` is a manual per-weapon checkbox (toggleable inline in `InventoryTab.js`'s item row) — the engine has no model of class weapon-proficiency lists and PDF import doesn't export one, so this mirrors the existing `attuned` philosophy of "the player just knows and tells the app." Extra Attack counting **was** later added (see below) — this paragraph used to say it was deliberately out of scope; that's no longer true, don't trust old comments in `ActionEconomyTab.js` over this file if they ever drift. Still out of scope: no separate Haste-bucket attack trigger (a hasted character's bonus attack isn't wired up — only the normal Action bucket).

### Extra Attack counting and Unarmed Strike — and the standing rule this established: compute from class/level, never hardcode to one character

`turnUsed` (in `CharacterContext.js`) gained an `Attacks` counter alongside the existing `Action`/`Bonus Action`/`Reaction` booleans. `maxAttacksForCharacter(features)` in `utils/dnd.js` detects an "Extra Attack" feature by name substring (same approach as Divine Smite/Sorcery Points detection) and returns 2 instead of 1 — the Action bucket itself only marks used once `Attacks` reaches that max, so a martial character correctly gets two swings logged before anything else sharing the Action bucket dims. `WeaponAttackModal.js` has a `thisAttackCounted` local flag so rolling Attack and rolling Damage for the *same* swing only counts once, and an `attacksExhausted` escape hatch (`&& !thisAttackCounted`) so finishing the *current* swing's damage roll never gets blocked by the live attack counter the same click just incremented — this exact bug shipped once (Roll Attack instantly locked out Roll Damage when it was the character's last attack) before being caught and fixed.

Unarmed Strike is always shown as a row in the WEAPONS section — RAW always allows it, with or without a magic item involved — built as a virtual, non-inventory "weapon" object (`unarmedStrike` in `ActionEconomyTab.js`; `WeaponAttackModal.js` takes it via a `weaponOverride` prop instead of `itemIndex`, since it has no real inventory row). Getting this right for a level-11 Monk surfaced the actual standing principle this section exists to document: **a fix driven by one character's imported sheet must be computed dynamically from that character's own class/level data, not hardcoded to the specific number that character happened to have.** Concretely, Unarmed Strike's damage was originally a hardcoded flat `"1" + STR mod` for every character — wrong for any Monk, whose Martial Arts feature both scales the die by **Monk class level specifically** (1d4 at 1-4, 1d6 at 5-10, 1d8 at 11-16, 1d10 at 17-20 — not total character level, which matters for a multiclass like the Monk 11/Barbarian 2 sheet that surfaced this) and lets the attack use DEX instead of STR. `martialArtsDie(classNameRaw, totalLevel)` in `utils/dnd.js` computes this live from `parseClassLevels` (the same multiclass-string parser spell-list filtering already used) with a same-named-single-class fallback for manually-built characters, and the DEX-or-STR choice is implemented by just tagging the virtual weapon with `properties: ['Finesse']` — reusing `weaponAbilityMod`'s existing "better of STR/DEX" logic instead of adding a parallel ability-choice path. **When a future per-character bug report turns out to be class-feature-driven** (a different class's resource, a different scaling rule), the fix belongs in a `class_name`/level-aware helper function like this one, not a name/value check against the one character whose sheet exposed it.

Items can grant a bonus to Unarmed Strike specifically via two fields that mirror a weapon's own `bonus_damage_dice`/`bonus_damage_type`: `grants_unarmed_bonus: true` plus `unarmed_bonus_damage_dice`/`unarmed_bonus_damage_type` (set via `AddItemModal.js`'s "Boosts Unarmed Strikes" checkbox, for non-weapon items like gauntlets) — `ActionEconomyTab.js` looks for any equipped+attuned item with this flag and folds its bonus dice into the virtual `unarmedStrike` object, tagging the row with `(Item Name)` so it's clear what's granting it (a real gap found in practice: a passive item with no charges had *no* visible presence anywhere in the AE tab before this, the ITEMS section only ever looked for `it.charges`). A heal-or-advantage choice (a 2024-style "deal bonus damage, then heal that much OR gain advantage on your next roll") is available as `bonus_heal_or_advantage` on a regular weapon's own bonus-damage fields too — deliberately the *same* flag name space as a weapon's bonus damage, not unarmed-strike-specific, after a real item (gloves set up as a plain Weapon, attacked with directly rather than through the unarmed-strike path) showed the unarmed-only version was too narrow. The prompt appears both after a digital damage roll and from "I'll roll in person" (manual entry of the rolled amount, no auto-rolling) — covers a player who wants to physically roll every die but still have the app track the resulting HP/advantage.

`passiveItems` in `ActionEconomyTab.js` is the general fix for "a passive item with no charges has nowhere to show itself": any equipped+attuned item with either `buffs` or `grants_unarmed_bonus` now appears in the ITEMS section listing its active effects via `formatItemBuff`, even with nothing to spend/recharge.

### Fighting Styles — found via a real mismatch against Barry's D&D Beyond sheet (Dueling's +2 damage was missing entirely)

Barry's Vicious Longsword showed `1d8 +9 Slashing` in the app but `1d8+11 Slashing` on his D&D Beyond sheet (the parenthetical `(1d10+9)` on the DDB sheet — the Versatile two-handed mode, which RAW correctly loses Dueling's bonus — was the tell: the gap is exactly +2 and disappears when wielding two-handed, which is Dueling's signature, not a magic-item or ability-score discrepancy). Audited every call site of `weaponAbilityMod`/`weaponItemBonus` (there's exactly one — `WeaponAttackModal.js` — confirmed via grep, so this wasn't split across multiple duplicated calculations) and confirmed **Fighting Styles were not modeled anywhere in the app at all**, for any character, even on the manually-created-character engine path (`content_packs.py`'s `"Fighting Style"` feature has always been a placeholder — `"Choose a fighting style specialty"` — with no field for *which* style was picked).

Fixed with `fightingStyleBonus(features, weapon)` in `utils/dnd.js`, detecting by feature-name substring (same approach as Extra Attack/Sorcery Points/Martial Arts) rather than adding a new "which style did you choose" setting — a PDF-imported sheet already has a feature literally named after the style (e.g. `"Dueling"`). Scoped to the two styles whose bonus is a flat, always-on number computable purely from data this app already has: **Dueling** (+2 damage, melee weapon held in one hand — reuses the *existing* "Wielding two-handed" checkbox's state as the one-hand/two-hand signal, since that's literally the same fact Dueling's "one hand" requirement cares about, so no new UI was needed) and **Archery** (+2 to attack rolls with a Ranged weapon). Wired into `WeaponAttackModal.js`'s `attackMod`/`buildDamage()` — which is the single source every roll, reroll, and the static preview line at the top of the modal all read from, so fixing it there fixed it everywhere at once — and explicitly excluded for the virtual Unarmed Strike weapon (RAW: Dueling/Archery require an actual weapon).

**Deliberately not modeled** (situational/manual riders that don't reduce to a flat number from existing data): Great Weapon Fighting's reroll-1s-and-2s, Two-Weapon Fighting's off-hand ability modifier (the app has no off-hand/dual-wield attack flow at all), Defense's +1 AC (would need to feed `CharacterHeader.js`'s AC calc, a different code path — not attempted in this pass since the reported bug was specifically about weapon damage), Protection's reaction, Blind Fighting, Superior Technique. Same "track what's computable, the player applies the rest" philosophy as conditions/exhaustion. If one of these comes up for a real character, same fix pattern applies: a `class_name`/feature-aware helper, not a hardcoded check for the one character whose sheet exposed it.

Separately, Barry's Vicious Longsword's bonus damage was typed in as `2d6 Piercing` when it should be `2d6 Slashing` (Vicious's rider deals the weapon's own damage type per RAW) — traced end-to-end and confirmed this is **not** a code bug: nothing in the PDF importer or static item DB sets `bonus_damage_dice`/`bonus_damage_type` for this item (there's no automatic source for a player-specific named weapon like this), so it was manually typed into `AddItemModal.js`'s "Bonus Damage Type" field at some point. No code fix needed — the player just needs to reopen the item, change Bonus Damage Type to blank (`(same as weapon)`, which auto-resolves to the weapon's own type) or explicitly `Slashing`, and save. Couldn't fix this directly since it's only in the live production Postgres DB, not anything reachable from this sandbox.

### Hybrid Form (Blood Hunter, Order of the Lycan) — same "compute from class level, never hardcode" rule Fighting Styles/Martial Arts already established

The owner asked how to natively track a Blood Hunter's Hybrid Transformation — RAW for Order of the Lycan has **no separate "full beast form"**; the controlled curse replaces normal Monster Manual lycanthropy rules for this subclass, and the only mechanical transformation is Hybrid Form (same stats/gear/speech, plus a bundle of always-on bonuses while active). A toggle button (🐺 Transform/Revert, Action Economy toolbar, only shown if the character has a feature whose name includes "Hybrid Transformation") flips `HYBRID_FORM_EFFECT` ('Hybrid Form') in/out of `tracker_data.active_effects` — same active-effects-driven design Haste already established, not a new parallel mechanism. `bloodHunterLevel`/`hybridFormStats` in `utils/dnd.js` compute every level-scaled number (melee damage bonus, unarmed die, unarmed attack bonus, max uses) live from the character's Blood Hunter class level via `parseClassLevels` (multiclass-aware, same as `martialArtsDie`), not hardcoded to any one character.

While active: `hybridFormBuffItems(td)` synthesizes a virtual always-equipped "item" (`+1 AC`, resistance to nonmagical/nonsilvered B/P/S, advantage on STR saves) folded into the same `buffItems` array feats/race already use in `CharacterHeader.js` — so AC, resistance chips, and the advantage chip all apply automatically with zero new rendering code, same trick `featBuffItems` already established. Unarmed Strike's virtual weapon (`ActionEconomyTab.js`) gets its damage die overridden to scale 1d6→1d8 at 11th level (reusing the existing Finesse/DEX-or-STR mechanism Martial Arts already uses) plus its own `weapon_attack_modifier`/`weapon_damage_modifier` buffs for Predatory Strikes/Improved Predatory Strikes. Feral Might's melee damage bonus is **not** unarmed-only per RAW ("bonus to melee damage rolls") — a separate `hybridFormWeaponBonus(td, classNameRaw, totalLevel, weapon)` applies it to any real wielded melee weapon too (skipped for the virtual Unarmed Strike specifically, since that one already carries its own copy via `buffs` and double-counting would otherwise result), wired into `WeaponAttackModal.js` alongside `featWeaponBonus`/`fightingStyleBonus`.

Charge tracking (1 use at 3rd, 2 at 11th, unlimited at 18th) degrades gracefully for a PDF-imported sheet with no numeric Uses printed — same "no charge to spend, button still works" precedent already established for other passive imported features — and only decrements `features['Hybrid Transformation'].current` when a real `max > 0` is actually tracked. Transforming also auto-switches `tracker_data.active_companion` to `companion2` if the player has set one up as the Hybrid form's separate ability list (Settings → Track a Companion — this was the *existing* answer to "Blood Hunters need two stat blocks," see the Companion tracking notes in earlier sessions), so both halves of that setup move together with one click instead of needing the player to flip the companion pill separately. Bloodlust shows as a header reminder chip (DC 8 WIS save or attack nearest creature) only when transformed **and** at/under half HP — not auto-rolled, same "track it, the player applies it" philosophy as conditions/exhaustion everywhere else in this app.

**Deliberately out of scope for this pass**: Crimson Rite application (no rite-tracking mechanic exists anywhere in this app yet, for any Blood Hunter order); auto-reverting Hybrid Form at 0 HP/unconsciousness (nothing else in this app triggers an automatic state change off HP either); a cosmetic non-Lycan "full beast form" some tables homebrew on top of RAW (would need its own design decision if ever requested — RAW itself doesn't have one). **Not yet visually verified in a browser** (no Node/browser access in this sandbox) — next session should transform a Lycan Blood Hunter character, confirm the AC/resistance/advantage chips appear, roll an Unarmed Strike and a real melee weapon attack to confirm the damage bonuses apply, and drop below half HP to confirm the Bloodlust reminder shows.

### Class resources that scale with level (Sorcery Points, Divine Smite) — a recurring data-completeness gap, not a missing mechanism

`character_engine.py`'s `build_tracker_data()` has a "Dynamic use counts" block that overrides `content_packs.py`'s placeholder `max: 0` with a real computed number for several features (Rage uses, Divine Sense, Lay on Hands, Bardic Inspiration, Ki Points, Action Surge, Indomitable) — the mechanism for "a class resource whose max depends on level/ability score" already existed. Sorcerer's "Font of Magic" (Sorcery Points) just wasn't in that list, so it sat at a permanent `max: 0`, purely descriptive, never actually trackable, the same way Divine Smite's entry was. Fixed by adding a `Sorcerer` block: `sorcery_points = level` from 2nd level on (RAW), same `features[name]["max"]` override pattern as everything else. **What this does *not* do**: model what each individual Metamagic option mechanically does (Twinned, Quickened, etc. all have different, bespoke effects) — Sorcery Points is tracked as a plain spendable number, same "track the resource, the player applies RAW knowledge themselves" philosophy already used for conditions/exhaustion having no enforced effects.

Divine Smite is structurally different from a points pool — it doesn't have its own charge count at all, it just spends a *spell slot* (already tracked) on a melee hit for bonus radiant damage. Modeled in `WeaponAttackModal.js` instead of as a tracked feature: if `tracker_data.features['Divine Smite']` exists (key presence only, not max) and the weapon isn't Ranged, a checkbox + slot-level dropdown appears next to "Roll Damage?"; rolling computes `2d8 + 1d8 per slot level above 1st (capped at 5d8)` (+1d8 more vs undead/fiend, a manual checkbox — the app has no creature-type-of-target concept to check automatically) and spends the slot via the existing `useSlot()`. The slot is only spent on the *first* roll, not on "Reroll" (`rerollDamage()` reuses the already-rolled smite's dice spec rather than recomputing/re-spending) — getting this wrong was the actual bug caught while building it: the original wiring would have spent a fresh slot on every reroll.

**This only fixes the *mechanism*, not every individual character's data.** `build_tracker_data()` is the manually-created-character path; PDF-imported characters (the owner's own Syric Nightbloom, and any multiclass Paladin/Sorcerer like the example that prompted this) don't go through it at all, so a PDF-imported character's "Font of Magic" still parses in at `max: 0` regardless of this fix, with no automatic resync-time recomputation (`pdf_import.py` doesn't know any class's scaling formulas). The immediate workaround for an existing PDF-imported character: delete the inert imported "Font of Magic" feature (Tracker tab's × button — PDF-imported features are deletable) and re-add it via "+ Custom" as a plain feature named e.g. "Sorcery Points" with Uses set to the character's actual sorcerer level, long rest. Divine Smite needs no such workaround — it already works for any character whose sheet has a feature literally named "Divine Smite", since the check is name-based, not engine-path-based.

Also fixed in the same pass: `level_up_character()`'s feature-merge had a real bug — `_merge_features_for_level_up()` computed `min(old_current, new_max)`, which only ever *caps* the old current at a higher max without ever granting the newly-gained capacity as available (2/2 Sorcery Points used 1, level up grants a 3rd point, should read 2/3 available — the old formula left it at 1/3). Fixed to compute "spent" from the old state and subtract that from the new max, the same logic `_merge_spell_slots_for_level_up()` already used correctly.

**Known remaining gap, explicitly out of scope for this pass**: other classes' scaling resources that aren't in the "Dynamic use counts" block at all yet (Battle Master Superiority Dice, Monk's various Ki-spending abilities beyond the point pool itself, Wild Shape uses, etc.) — same fix pattern applies whenever one of these comes up for a real character, but auditing all 12 classes' subclass resources wasn't attempted here.

### Header status row: active effects, conditions, exhaustion

`tracker_data.active_effects` (Hasted, Bardic Inspiration, Rage, etc.) and `addActiveEffect`/`removeActiveEffect` already existed and worked, but only surfaced on the Tracker tab — easy to forget mid-combat. `tracker_data.conditions` existed as data with a `GET /api/content/conditions` route already serving the 15 SRD conditions, but had no add/remove UI anywhere (`TrackerTab.js` only ever rendered it read-only) — `addCondition`/`removeCondition` were added to `CharacterContext.js` (after `saveTrackerData`, per the TDZ rule below) to close that gap. `tracker_data.exhaustion` didn't exist at all; it's now a plain `0–6` integer, stepped via the existing `PMStat` component (no new component needed — same pattern as Temp HP) and clamped client-side, since nothing else in this app mechanically enforces condition/exhaustion effects and this is consistent with that.

All three now live in a new row in `CharacterHeader.js`, inserted between the main stat row and the trait-chips row — deliberately a separate row rather than folded into trait chips, since trait chips are static/hover-only (resistances, immunities) while this row is interactive (click-to-remove effect chips, a stepper, a button that opens `ConditionsModal.js`), and mixing those two interaction models in one row would make it unclear at a glance what's clickable. `ConditionsModal.js` excludes "Exhaustion" from its checklist (it has the header stepper instead) and shows each condition's description inline rather than requiring a second click-through, since there are only 14 and players are usually checking "what does this actually do" in the moment. Adding a new effect from the header uses a small `CoinCalculator`-style popover (`EffectAdder` in `CharacterHeader.js`) rather than a full modal, since typing one word like "Hasted" doesn't need more ceremony than that.

### Concentration tracker, ported from the Pi tracker's `action_economy_subtab.py`

The owner's old Raspberry Pi/Tkinter tracker (outside this repo, under `Desktop\Wearables\Syric Arcane Controller\` — the live working copy with a `.git` folder, *not* the dated `Backup` folders, which lag behind) has a concentration feature this one ported directly: 1-2 "CON" slots, auto-filled the moment a concentration spell is actually cast (not a manual "type a spell name" field — there is no manual-entry path in either tracker), with a prompt to replace a slot only when all active slots are already full. `tracker_data.concentration = {slots: [{spell, level}, {spell, level}]}` (always 2 slots stored; `concentrationSlotCount(items)` in `utils/dnd.js` decides how many are *active* — 1 by default, 2 if an equipped+attuned item's description reads like it grants a second slot, generalized from the Pi's hardcoded "Black Veil" item-name check since this app has no fixed character to special-case). `spell.concentration` is already a clean boolean in `spells.json`, unlike the Pi's fuzzy duration-text detection, so no port needed there.

The hook lives in `SpellDetailModal.js`'s `doCast()` — the single universal cast path used by every "Cast" button in the app (header, Spells tab, Action Economy's CAST button, item-granted spells) — right after a slot/charge/cantrip cast succeeds: `tryTrackConcentration()` auto-fills an open slot silently, or sets `concPrompt` state to show a "replace which slot?" footer view (mirroring the existing damage-roll footer pattern) when none are free. `ConcentrationModal.js`, opened from a CON badge in `ActionEconomyTab.js`'s toolbar, is view/drop-only — same as the Pi version, the *only* way a slot gets a name is by casting that spell through the app. Dropping a tracked "Haste" specifically warns about the lethargy rule (5e RAW: the target can't act until after its next turn), ported verbatim from the Pi's equivalent warning. Not ported: the Pi's automatic linkage between concentrating on Haste and the Haste action-economy bucket (`isHasted` in `ActionEconomyTab.js` still only checks `active_effects.includes('Hasted')` independently) — worth unifying later but out of scope for this pass.

Each concentration slot also carries a `target: 'self'|'ally'|undefined` tag (`tracker_data.concentration.slots[i].target`), needed specifically for Haste: the end-of-effect cleanup (remove the Hasted active_effect, apply Lethargic) should only ever touch the *caster's own* state when Haste was cast on **self** — casting it on an ally and later dropping it must leave the caster's own effects/conditions alone (they were never Hasted) and instead just show an info-only reminder about the ally. The tricky part is *when* the target is known: the concentration slot is filled the instant the cast succeeds (`tryTrackConcentration`), but the Self/Ally choice (`SELF_TARGET_EFFECTS`/`awaitingTarget`) isn't resolved until afterward — so the slot is filled with `target: undefined` first, then `chooseTarget()` patches it in once the player actually picks. `CharacterContext.js`'s `replaceConcentration(idx, newSpellName, newLevel, newTarget)` is the **one** place a slot is ever cleared or overwritten (covers both `ConcentrationModal`'s plain Drop and `SpellDetailModal`'s "concentration full, replace which spell?" prompt) — it reads whatever spell/target *used to be* in that slot before deciding whether Haste-cleanup applies, returning `{wasSelfHaste, wasAllyHaste}` so each caller can show the right message. This consolidation is itself a bug fix: before it, `ConcentrationModal`'s Drop button had its own inline cleanup and `SpellDetailModal`'s replace-prompt had none at all, so casting Polymorph to bump Haste out of a full concentration slot silently skipped the AC/effect/condition cleanup entirely.

### Character level-up (v1 framework) and the base-stat editor built on top of it

Settings → "✏️ Edit Character" opens `CharacterEditorModal.js`. It started as a level-up-only button; it now also has Identity (Name/Race/Class/Subclass/Level as plain editable text — Level here is a raw override independent of the engine-driven Level Up button below, which is what a PDF-imported/multiclass character has to use instead), Ability Scores (6 number inputs, same `ability_scores` PUT the header's click-to-edit boxes already used), and Saving Throw / Skill Proficiency checkboxes (`tracker_data.save_proficiencies`/`skill_proficiencies`, plus a new per-skill Expertise checkbox writing `tracker_data.skill_expertise` — previously `calcSkills` was always called with a hardcoded empty expertise array in `SkillsModal.js`, so Expertise had no effect anywhere in the app even though the calculation function already supported it). Before this, ability scores had a header click-to-edit box each but save/skill proficiencies had **no editing UI anywhere** — manually-created characters never even got `skill_proficiencies` populated by the engine, only PDF import parsed it from the sheet. The save-throw checkboxes default-check whatever `SAVE_PROFS[className]` would already imply, so a character who's always relied on that fallback doesn't see an empty checklist the first time they open this.

The original level-up mechanism is unchanged: `POST /characters/<id>/level_up` calls `character_engine.py`'s `level_up_character()`, which re-runs `build_tracker_data`/`build_ae_data` at the new level and **merges** the result with the character's existing live state, mirroring the merge philosophy `pdf_import.py`'s resync already established: current HP/hit-dice/feature-charge counts are preserved and only adjusted by however much the new level actually grants (HP current goes up by the same amount max did, not a full heal; an already-spent feature use stays spent unless the new max is smaller), and any `ae_data` entry that isn't `source_type === "class"` (i.e. anything added via +Custom/Browse Feats) is kept untouched rather than being clobbered by the freshly-rebuilt class-feature list.

This single-class path is preserved byte-for-byte and still the one a manually-created character with a clean `class_name` actually exercises. **PDF-imported and multiclass characters now have a real second path** — see "Multiclass leveling engine and the structured Class editor" below, which superseded the old hard 400 ("this means the owner's own PDF-imported character can't use this yet") with an actual multiclass-aware engine. Read that section before assuming level-up still doesn't work for Syric/Barry-style characters.

### Multiclass leveling engine and the structured Class editor

`tracker_data.classes` (`[{class_name, level, subclass}]`) is the new structured, multiclass-aware source of truth for leveling — separate from (and additive to) the original single-class engine above, which keeps working unchanged for any character that's never touched this. `backend/engine/multiclass_engine.py` adds: `infer_classes(class_name_raw, level)` to best-effort-parse a decorated/multiclass `class_name` like `"Wizard 13"` or `"Paladin 6 / Sorcerer 6"` (returns `None` if any part is unrecognized, so the caller asks the player to confirm rather than silently guessing wrong); `compute_multiclass_hp()` for a fresh build (only the very first level overall gets the max-hit-die bonus); the RAW multiclass spellcaster slot table (full casters count whole levels, half casters count half rounded down, the combined "effective caster level" indexes into the same single-class "full" progression table) plus a separate `pact_slots` pool for any Warlock levels (restored on short *or* long rest in `rest_engine.py`, independent of the existing single-class Warlock convention of storing pact slots directly in `spell_slots`); per-class ASI levels (4/8/12/16/19, with Fighter's extra 6/14 and Rogue's extra 10) and RAW subclass-choice levels; and auto-granting `class_features.json` entries (with a real Action Economy row via `_add_ae_entries_for_features`, not just a `tracker_data.features` entry) for both a newly-leveled class and a freshly-picked subclass.

**A critical correctness fix inside `level_up_one_class()`**: HP gain is computed *incrementally* (just the one new level's average-roll amount, added to whatever max HP was already recorded) — **not** by recomputing the character's entire HP from scratch the way it might seem natural to. An earlier version of this function did recompute from scratch via a `compute_multiclass_hp()` call on every level-up, which is fine for a character built entirely by this engine but breaks badly for any PDF-imported character whose real HP doesn't match the generic average formula (rolled HP, a CON score that changed after some levels, etc.) — the recomputed total essentially never matched, and when it came out *lower* than the real recorded max, the `max(0, …)` floor reported "+0 gained" while silently shrinking max HP to the wrong, lower value. This is exactly the kind of bug this codebase has hit before with "recompute from scratch when you should adjust incrementally" — keep the incremental pattern if this function is touched again.

`POST /characters/<id>/level_up` now branches three ways: the original single-class path (unchanged); a multiclass path that needs `tracker_data.classes` confirmed first (`needs_class_confirmation`, with `infer_classes()`'s best-effort guess attached so the frontend can pre-fill it); and once confirmed, a `needs_leveling_class_choice` prompt for which class is gaining the level if there's more than one and the caller didn't already say. New `POST /classes/subclass` and `POST /asi` routes apply a subclass pick or an Ability Score Improvement as separate, explicit actions instead of level-up silently auto-applying everything. `GET /class_status` is a read-only check (no level-up side effect) that backs a proactive "Confirm Classes" banner on `GameView.js` for any character whose `class_name` doesn't resolve cleanly — closes the "notify the user to select a class and subclass if the import wasn't done correctly" gap.

A single-step undo: `POST /level_up` snapshots everything it's about to touch (`tracker_data`/`spell_data`/`ae_data`/`level`/`class_name`/`subclass`/`ability_scores`) into `tracker_data._level_up_snapshot` before applying anything, overwriting any prior snapshot (one level-up's worth of undo, not a history stack) — survives any follow-up subclass/ASI calls in the same flow since those only ever shallow-copy `tracker_data`. `POST /rollback_level_up` restores from it and clears it. Surfaced as an immediate "Undo, I'll check first" option right on the level-up success screen, and a persistent "Roll Back to Level N" button in the character editor that stays available even after closing the popup — players explicitly like to preview a level-up and back out if they don't like the result.

`LevelUpFlowModal.js` is a state machine driven entirely by `POST /level_up`'s responses: confirm classes → choose leveling class → (if `tracker_data.hp.calc_mode === 'rolled'`) roll/enter Hit Die HP → choose subclass(es) → choose ASI or skip for a feat → done. `mode='confirm_classes'` (used by the GameView banner) skips straight to the class-confirmation step with no level-up side effect. `initialLevelingClass` skips the chooser step entirely when the caller already knows which class is leveling. Feat selection at an ASI level is deliberately *not* reimplemented here — the player is pointed at the existing "Browse Feats" flow on the Feats/Attunement tab instead of duplicating that attach logic a second time.

**HP calculation method**: `tracker_data.hp.calc_mode` (`'fixed'` default, unchanged behavior, or `'rolled'`, toggled in `HPModal.js`) — Rolled inserts a step right after a successful level-up showing the leveling class's Hit Die, a "Roll 1d{N}" button (`rollDie` from `dnd.js`), or a manual-entry field for what was rolled in person; the difference from the already-applied average is applied as a delta to both max and current HP (same "adjust by the change, never blind-overwrite" pattern the HP Max Override fix below uses), not a blind overwrite.

`CharacterEditorModal.js`'s Identity section no longer has a free-text Class/Subclass/Level fallback at all — the structured Class 1/Class 2 rows (each with Class/Level/Subclass fields, its own "Level Up [ClassName]" button, a locked-to-that-class-and-subclass "📖 Preview" button capped at the highest level that class could actually reach given the character's other committed levels, and "+ Add Another Class") are bootstrapped from the character's existing `class_name` the moment the class list loads, filling in a `class_name` only where it's a confirmed exact match (an unrecognized part is left blank for the player to pick, never silently guessed wrong) — `parseClassLevels` from `dnd.js` does the parsing. **The Classes section saves itself immediately on every change** (Class/Subclass selects and Add/Remove commit right away; the Level number commits on blur to avoid a keystroke race) instead of being part of the Cancel/Save Changes draft form below it — this is a deliberate fix for a real bug: editing one row then clicking a *different* row's Level Up/Subclass button used to force-commit every row's draft state as a side effect (those actions needed the server in sync first), so clicking Cancel afterward looked like it silently failed to revert anything. `saveClasses()` also keeps the Character model's own `class_name`/`subclass`/`level` display columns (used everywhere else in the app — header, character list) in sync on every class edit, since they're a derived display of `tracker_data.classes`, not a separate source of truth. `saveAll()`/Cancel now only ever cover Name/Race/Ability Scores/Proficiencies.

`ClassFeatureBrowserModal.js` (already existed as a searchable add-to-character library) gained `lockedClass`/`lockedSubclass`/`maxLevel` props for the read-only preview use case (`onAdd` omitted hides the Add button entirely) and is twice as wide.

**Duplicate Character**: `POST /characters/<id>/duplicate` clones name/ability_scores/tracker_data/spell_data/ae_data/notes/source_pdf into a brand new, fully independent row (named "X (Copy)") — a "📋 Duplicate" button on the character list. Lets a player freely experiment with a level-up or build change on the copy without any risk to the original; explicitly requested specifically because of how much the level-up engine now lets a player change.

### Class resources now scale automatically across BOTH leveling engines, plus an on-demand fix for already-existing characters

`character_engine.py`'s "Dynamic use counts" block (Rage uses, Divine Sense, Lay on Hands, Bardic Inspiration, Ki Points, Action Surge, Indomitable, Sorcery Points) only ever ran for the single-class manual-build path. The multiclass/PDF-import leveling path (`level_up_one_class` above) granted new class **features** correctly via `class_features.json` on every level-up, but never recalculated any of these formulas — so a multiclass or PDF-imported character's Rage uses (etc.) stayed wherever the PDF parser happened to leave them (often `max: 0`) forever, even across real level-ups. This is the gap the owner specifically called out with Barbarian Rage and Fighter Action Surge as examples.

Extracted into a shared `apply_dynamic_feature_maxes(features, class_name, level, ability_scores)` in `character_engine.py`, meant to run on a **freshly-built** features dict (current == max for everything) before merging — not directly on a live character's already-in-play features — so `_merge_features_for_level_up`'s existing "preserve spent, grant the gain" logic stays the *one* place spent-amount preservation happens, for every caller. `level` is always the class's *own* level (e.g. Barbarian level within a Barbarian 5/Fighter 3 build), never total character level — matches RAW, where every one of these scales off a single class's own table. Both `build_tracker_data` (single-class) and `level_up_one_class` (multiclass) now call it the same way: build fresh features → `apply_dynamic_feature_maxes` → merge.

**On-demand fix for a character whose resources were already wrong before this existed**: `recalculate_class_resources(tracker_data, ae_data, classes, ability_scores)` in `multiclass_engine.py` + `POST /characters/<id>/recalculate_class_resources` re-syncs *every* current class (works for a plain single-class character too — treated as a one-entry classes list, same underlying function, no second engine) at its *current* level: rebuilds that class's full base+subclass feature set (auto-granting anything missing, same merge semantics as a real level-up) and recalculates its dynamic maxes — never refills an already-spent resource, never lowers a max below what's already in use. `CharacterEditorModal.js`'s Classes section has a "🔄 Recalculate Class Resources" button wired to this — the practical fix for a PDF-imported character who's been stuck with broken Rage/Action Surge/Sorcery Points/etc. since import, without needing an artificial level-up-then-rollback workaround.

Verified via ad hoc script (no test suite in this repo, per the standing convention) before shipping: single-class Barbarian L5 Rage came out 3/3 (correct per the RAW table); Fighter L2→L17 Action Surge went 1/1→2/2 and Indomitable came out 3/3; a stale multiclass Barbarian 4/Fighter 1 build with Rage stuck at 0/0 (simulating a bad PDF import) correctly recalculated to 3/3 via the new on-demand route; and leveling a Barbarian 4→5 with 1/3 Rage already spent correctly preserved the spent amount (1/3, not refilled to 3/3) since the RAW table value doesn't change between levels 4 and 5.

**Known remaining gap, same as before this session**: other classes' scaling resources still aren't in this list at all (Battle Master Superiority Dice, Monk's various Ki-spending abilities beyond the point pool itself, Wild Shape uses, etc.) — same fix pattern applies (add a branch to `apply_dynamic_feature_maxes`) whenever one of these comes up for a real character. **Not yet visually verified in a browser** (no Node/browser access in this sandbox) — next session should click the new Recalculate button on a real multiclass or PDF-imported character and confirm Rage/Action Surge/etc. actually update on the Feats/Attunement tab.

**Known, documented limitations of the multiclass engine**: third-caster subclasses (Eldritch Knight, Arcane Trickster) aren't counted toward the multiclass spell slot table, since this app has no subclass-mechanics model for base classes; multiclass hit dice of different sizes per class share one `die_size` field (the count is still tracked correctly, just not enforced per-die-size on a rest); homebrew classes (Pugilist/Illrigger/Blood Hunter) default to a 3rd-level subclass choice since their real PHB-equivalent level isn't specified anywhere in this app's data; a player who adds a blank "+ Add Another Class" row, doesn't fill it in, then levels up a *different* row loses that blank row (it was never persisted, so the post-level-up resync from the server doesn't know about it) - minor, just re-add it.

### Sorcerer Points and Metamagic engine, plus a closure-staleness fix in `CharacterContext.js` that this surfaced

The Sorcerer feature is called `"Font of Magic (Sorcerer Points)"` everywhere in the engine now (`content_packs.py`, `character_engine.py`'s level-scaling block) — renamed from the bare "Font of Magic" so the resource is self-explanatory wherever it's listed. `TrackerTab.js`/`SpellDetailModal.js` detect it **by substring match** (`name.toLowerCase().includes('font of magic')`), not by exact string, specifically so this still works for older or PDF-imported characters whose feature happens to be named just "Font of Magic" — the detection doesn't care which exact spelling is on a given character's sheet.

`SorceryPointsModal.js` (opened via a "🔮 Manage" button next to that feature in `TrackerTab.js`) has two tabs: **Flexible Casting** (RAW: convert points into a spell slot — costs more than the slot's level, table in `SORCERY_POINTS_TO_SLOT_COST` in `utils/dnd.js`, capped at a 5th-level slot, created slots vanish on the next long rest if unused — or convert a slot into points, gaining exactly the slot's level; both mark the Bonus Action bucket consumed when in initiative) and **Metamagic** (a checklist of all 8 PHB options plus Tasha's Cauldron's Seeking Spell and Transmuted Spell — `METAMAGIC_OPTIONS` in `utils/dnd.js`, paraphrased in the app's own words rather than the printed rules text — saved to `tracker_data.metamagic_known`). `metamagicCost(name, spell)` handles Twinned Spell's variable cost (equal to the spell's level, minimum 1) versus everyone else's flat cost.

Applying a known Metamagic option happens in `SpellDetailModal.js`'s cast flow: a dropdown appears above the Cast button whenever `metamagic_known` is non-empty (skipped for `chargeMode`/item-granted casts — kept to the common case of casting your own spell), showing each option's cost and disabling ones the character can't currently afford; the chosen option's point cost is deducted as part of `doCast()`, and its paraphrased effect text is shown so the player can apply it themselves (the app doesn't, say, actually double a Twinned Spell's targets for you — same "track the resource, the player applies RAW" philosophy already used for conditions/exhaustion).

**Building this surfaced a real, pre-existing bug**, not just a new-code risk: `doCast()` already called `useSlot()` *then* (via `tryTrackConcentration()`) `setConcentration()` for any concentration spell cast with a real slot. `setConcentration`'s old implementation built its save payload by spreading `character.tracker_data` from `SpellDetailModal`'s own closure — fine in isolation, but that closure was fixed at the render *before* the click, and `useSlot()`'s update happens *during* the same `doCast()` call. JS closures don't retroactively pick up newer bindings mid-execution, so `setConcentration`'s spread was silently built from the *pre-cast* `tracker_data` — meaning its save would overwrite the just-decremented spell slot back to its pre-cast value. This was apparently never noticed because nobody had compared the *actual persisted* slot count after casting a concentration spell against what the UI showed mid-session. Fixed at the root in `CharacterContext.js`: a `characterRef` (a plain `useRef`, kept in sync **synchronously** inside a wrapped `setCharacter`, not via a `useEffect` — effects run after React's next commit, too late for a function that's already mid-`await`) that `setConcentration`/`setConcentrationTarget`/`replaceConcentration`/the new `spendFeatureCharges` (used for the Metamagic point deduction) all read from instead of the closure-captured `character`. This makes those four functions safe to call from any multi-step sequence regardless of what else in that sequence already changed `tracker_data`, without needing to chase down and fix every call site individually the way earlier sessions did for the Haste/concentration bugs.

### Beast summoning (v1) — `tracker_data.active_creatures`

First slice of the larger unified summon/polymorph/wild-shape system (full spec saved to memory, not yet built in full). `BestiaryTab.js` now reads/writes `tracker_data.active_creatures` via `useCharacter()` (it was previously a pure read-only browser with no character context at all). `MonsterDetailModal.js` grew an optional `onSummon` prop — when passed, its footer shows a "🐉 Summon" button; `BestiaryTab.js` is the only caller that passes it (creatures viewed via the "Active Creatures" list reopen the same modal *without* `onSummon`, since re-summoning a creature you're already tracking doesn't make sense there). Summoning creates `{id, creature_name, instance_name, hp:{current,max}}` (`hp.max` from the monster's `hit_points` field) and appends it to the array; a second copy of the same creature gets `instance_name` suffixed `#2` etc. so duplicates are distinguishable in the list. The "Active Creatures" card (shown above the search card, only when non-empty) has inline HP ±/Dismiss and clicking a name re-opens the full stat block by looking the creature up again in the already-fetched `monsters` list by name — if that lookup ever fails (monster renamed/removed from the static DB since summoning), it alerts rather than silently doing nothing, so a stale entry doesn't look like a dead click.

**Deliberately not in this slice**: no polymorph/wild-shape self-form header takeover, no Polymorph-specific cast-flow integration in `SpellDetailModal.js`. See the `project-creature-tracking-spec` memory for the full agreed design of those pieces.

### Homebrew monster duplication — monsters join the `CustomContent` pattern

`backend/data/monsters.json` stays 100% canon/read-only; homebrew creatures now work exactly like homebrew spells/feats already did — `GET /content/monsters` merges the static file with `CustomContent` rows of `content_type="monster"` (shared across everyone, same as spells/feats, not scoped to creator), and there's now `POST` (create), `PUT /content/monsters/<custom_id>` (update), and `DELETE /content/monsters/<custom_id>`, none of which needed a migration since `content_type` was always a freeform string column. `MonsterDetailModal.js` shows a "📋 Duplicate" button on canon entries (`_source !== 'custom'`) and an "✏️ Edit" button on homebrew ones instead (mutually exclusive, never both) — `DuplicateMonsterModal.js` clones the full stat-block JSON under a new name (stripping `_custom_id`/`_source` first) and POSTs it.

`MonsterEditModal.js` is a deliberately pragmatic v1: a name field plus one big JSON textarea for everything else (`armor_class`, `hit_points`, `speed`, `ability_scores`, `actions`, etc.) rather than a purpose-built form per field — a monster stat block has far more structured sub-fields than a spell or feat, and building a dedicated editor for all of them wasn't worth the risk of shipping something untested this late in a long batch of changes. Same data shape `MonsterDetailModal.js` already renders, so anything valid typed in there displays correctly afterward. Includes a Delete button (with the standard `ConfirmModal`) to remove a homebrew entry entirely.

### No more native `window.alert`/`window.confirm` anywhere in the app

`InfoModal.js` (title + message + "Got it") and `ConfirmModal.js` (title + message + Cancel/Confirm, `danger` prop reddens the confirm button) are the standard replacements — every call site that used to pop the browser's native dialog (`InventoryTab.js` refresh feedback, `BestiaryTab.js` dismiss-creature confirm and stale-lookup alert, `ConcentrationModal.js`'s Haste-ended notice, `TrackerTab.js`'s remove-ability confirm) now uses one of these two instead, each via local `useState` + a result/error callback rather than a blocking synchronous call. **Any new alert/confirm anywhere in the app should use these, not `window.alert`/`window.confirm`** — the owner flagged the native dialogs as visually inconsistent with the rest of the UI ("odd popup").

### Session handling: 401s used to fail completely silently

Two related footguns, both fixed: (1) `JWT_ACCESS_TOKEN_EXPIRES` was never set in `app.py`, so flask-jwt-extended silently defaulted to **15 minutes** — every save after that point 401'd. (2) `utils/api.js`'s axios instance had no response interceptor at all, so a failed request (expired token, network blip, 500) just vanished with zero UI feedback, indistinguishable from data loss. Both are fixed now: token expiry is 30 days (fine for a single-user personal app), and the interceptor shows an alert on any failure, with a 401 specifically clearing the stored token and bouncing to the login screen with a clear message rather than just erroring. **If "something didn't save" alerts start recurring, suspect this layer first** before assuming a feature-specific bug — it's exactly what surfaced the (already-correct) Staff of the Magi data as "broken" once, when the real issue was a timed-out cross-reference request.

### Frontend: one context, and a sharp footgun in it

`context/CharacterContext.js` is the single source of truth — every tab/modal reads character state and mutates it through functions exposed here (`saveTrackerData`, `useSlot`, `useItemCharge`, `addActiveEffect`, `turnUsed`/`setTurnUsed`/`resetTurn` for the Action Economy turn-bucket UI state, etc.), each a thin wrapper that PUTs/POSTs and updates local state from the response. **Ordering matters**: these are all `useCallback`s defined as sequential `const`s in one function body, and several depend on `saveTrackerData` (it's in their dependency array). Defining a new helper *before* `saveTrackerData`'s own `const` line throws `ReferenceError` (temporal dead zone) the instant the provider renders — and since this provider wraps the entire app, that crash is a blank screen on *every* route, not just wherever the new function is used. This has happened more than once; when adding a new context function that calls `saveTrackerData`, place its declaration after `saveTrackerData`'s.

`GameView.js` renders `CharacterHeader` (always visible) plus one of six tabs (`ActionEconomyTab`, `TrackerTab` — labeled "Feats/Attunement" in the UI, "Tracker" before this session — `SpellsTab`, `InventoryTab` — labeled "Inventory" in the UI, "Items" before this session — `NotesTab`, `BestiaryTab`), each independently reading/writing `tracker_data` through the context.

`CharacterHeader.js` is a **three-column flex layout** (as of the header-restructure session — it was two columns before that, see git history if older context resurfaces): a left column (title row, then the main stat row: HP+spell-attack/DC stacked, Temp HP, AC, INIT, Prof, Inspiration, Attunement count, ability-score boxes with spell slot pips stacked below, then trait chips), a middle "status" column, and a right column (Currency + Saves/Skills/Traits/Rest buttons, Hit Dice under Rest). **None of the three columns use `flex:1`/stretch sizing** — each hugs its own content width, the right column gets `marginLeft:'auto'` to claim whatever space is left, and the outer row has `flexWrap:'wrap'` as a safety net for narrow viewports. This was a deliberate fix: the left column previously had `flex:1`, which stretched its box to fill leftover space without its *content* filling that space — so a visible gap appeared *inside* the left column's box, before the middle column (a separate sibling) even started, no matter where the middle column's own content began. The owner circled this exact gap in a screenshot more than once across sessions; `flex:1`-stretch-then-leftover-gap was the root mechanical cause, not the middle column's own width or position. The middle column's three pieces — Exhaustion `PMStat` stepper, active-effect word-chips + an `EffectAdder` "+" popover, active-condition word-chips + a Conditions counter-button that opens `ConditionsModal.js` — sit **side-by-side** as their own sub-columns (each effect/condition chip wraps under its own "Effects"/"Conditions" label) rather than stacked vertically, per explicit feedback that stacking made it unclear which chip belonged to which category. **When the owner circles empty space in a header screenshot, check for a `flex:1`/stretch container creating a phantom gap before assuming it's a positioning/width problem with whatever new content is being added** — this exact confusion repeated across multiple sessions before the actual mechanism was identified. The middle column exists specifically because conditions/effects kept landing in the wrong place across several iterations when they were nested inside the left or right column instead of being their own sibling — the owner explicitly asked for "hidden columns... where data can live" after circling the actual empty space in a screenshot, so **default to adding new header content as its own column-sibling, not nested inside an existing one**, and confirm placement against a fresh screenshot before assuming a text description of "the header" is still accurate (it has been redrawn many times). Currency editing is a popover numeric keypad (`CoinCalculator`) opened by clicking a coin value, not a free-text input — it does a relative +/- adjustment and Apply, not an absolute-value set. The Attunement count only turns red when it's an actual problem (over the 3-item cap, or under cap while an unattuned eligible item sits unused) — not just for being below 3/3.

### Class & subclass feature library (`backend/data/class_features.json`) — search and add, not auto-granted

The owner handed over a scraped copy of D&D Beyond's full Classes section (13 official classes + Artificer, plus three third-party homebrew classes — Pugilist, Illrigger, Blood Hunter — that also appeared in the doc) and asked for "all of these feats, abilities etc" to be searchable and addable to a character. Auditing `content_packs.py` against it surfaced the real gap: every class's *base* features were already reasonably filled in with paraphrased descriptions, but **zero subclasses existed anywhere in the engine, for any class** — "Choose your X subclass" has always been a placeholder line with no actual subclass mechanics behind it, across all 12 PHB classes.

Given the scale (every official subclass *and* every UA/Xanathar's/Tasha's variant the doc listed, per the owner's explicit choice to include everything rather than just core PHB), this was built as a new **searchable reference library**, the same architectural choice as feats/spells/monsters, rather than wiring subclass selection into the character-creation/level-up engine — that would additionally need a subclass-picker UI and auto-grant-at-the-right-level logic, a much bigger lift than what was asked for. `backend/data/class_features.json` holds ~1,260 entries (`class_name`, `subclass_name` — null for base-class features, `level`, `name`, `action`/`max_uses`/`rest_type` in the same shape `tracker_data.features` already uses, `description` — rewritten from scratch in plain mechanical language, never copied verbatim from the scraped D&D Beyond text for copyright reasons — `source`, `edition`). `engine/class_feature_data.py` (`get_all_class_features()`) serves it via `GET /api/content/class-features` (optional `class_name` filter), merged with `CustomContent` rows of `content_type="class_feature"` the same way every other shared library merges in homebrew. `ClassFeatureBrowserModal.js` (a "📚 Class Features" button on the Feats/Attunement tab, next to "Browse Feats") defaults its class filter to the character's own class via `parseClassLevels`, and "Add" attaches a feature to `ae_data`/`tracker_data.features` the same way `addFeatFromLibrary` already does for feats — browse-and-add, nothing is auto-granted on character creation or level-up, matching how feats already work.

Pugilist, Illrigger, and Blood Hunter (the three homebrew classes) also got real entries added to `content_packs.py`'s `CLASSES`/`SPELLCASTER_TYPE`/`SUBCLASSES` dicts (hit die, saves, base features) so they're selectable when manually creating a character — their subclasses live in the library only, same as everyone else's. Artificer (official WotC content) turned out to be missing from the engine entirely too, unrelated to the homebrew classes — added alongside them since it's the same gap.

### Feat modifiers (`buffs`) and the two-layer admin-edit/duplicate pattern, extended to feats

Feats previously had no way to express a mechanical stat bonus at all, and the only way to fix a wrong/badly-worded feat was deleting and retyping it. Extended the **exact** two-layer pattern items/spells/monsters already use: a `feat_override` `CustomContent` row corrects a canon `feats.json` entry in place (`PUT /content/feats/override`, matched by name, never touches the static file on disk), and a plain `feat` row is an independent duplicate/new homebrew feat. `FeatBrowserModal.js` gained a detail view (it was a flat add-only list before) with "✏️ Admin Edit"/"📋 Duplicate" for canon feats, "✏️ Edit"/"🗑️ Delete" for homebrew ones (already existed), and "↺ Revert" for an admin-edited canon entry — all driven through `CustomAbilityModal.js`'s new `editMode` prop (`'admin'` PUTs the override, `'duplicate'` POSTs a new row pre-named "X (Homebrew)", `'custom'` is the pre-existing edit-your-own-homebrew-feat path).

"Feat edit should give us the same level of control [as items]" meant feats needed `AddItemModal.js`'s Modifiers editor (AC/saves/spell attack-DC/ability score Set-To/Add-To/advantage-on-saves/damage resistance-immunity-vulnerability/condition-immunity) — that whole editor was extracted into a shared `ModifiersEditor.js` component (`buffs`/`onChange`/`allowWeapon` props) so AddItemModal, `CustomAbilityModal.js`, and `FeatureEditModal.js` (the "fix a feat already on my character" path — arguably the more common entry point than the library browser) all render and edit the identical buff shape instead of three copies drifting apart. **Deliberately excluded for feats**: `weapon_attack_modifier`/`weapon_damage_modifier` — those are inherently "this one weapon" concepts on items (`weaponItemBonus` in `dnd.js` reads only that weapon's own buffs, by design, so a +1 longsword's bonus never leaks onto other weapons); a feat granting a flat bonus to *all* weapon attacks has no equivalent character-wide pool to plug into yet, so offering the option without wiring it would just reproduce the exact "stored but never consumed" mistake item buffs made three separate times before being caught (see "Generalized item stat buffs" above).

Wiring a feat's buffs into actual play needed `featBuffItems(features)` in `dnd.js`: rather than teaching `computeItemBonuses`/`effectiveAbilityScores`/`calcSaves`/`calcSkills`/`getSpellcastingBlocks` a second, parallel aggregation for feats (and risking it drifting from the items version the way the "weapon_attack_modifier never pooled" class of bug kept happening), a feat with buffs gets synthesized into the exact shape an always-equipped, no-attunement-required item already takes (`{name, equipped: true, attunement: false, buffs}`) and concatenated onto the real `items` array at each of the half-dozen call sites (`CharacterHeader.js`, `SavesModal.js`, `SkillsModal.js`, `WeaponAttackModal.js`, `SpellDetailModal.js`, `SpellsTab.js`) — every existing consumer picks it up with zero changes of its own. A feat only gets a `tracker_data.features` entry at all (the only place `featBuffItems` can find its buffs) if it has uses, is a tuck-and-release/grants-a-spell ability, *or* now has buffs — a purely-passive, no-charge feat with a modifier needed that condition added in three places (`CustomAbilityModal.js`'s own-character-attach path, `TrackerTab.js`'s `addFeatFromLibrary`, and implicitly `FeatureEditModal.js` which always has a `features` entry to edit already).

### Flat JSON reference databases, loaded once at import time

`backend/data/spells.json` (412 spells), `backend/data/magic_items.json` (1,267 items), and `backend/data/class_features.json` (~1,260 class/subclass features, see above) are loaded module-level by `engine/spell_data.py` / `engine/item_data.py` / `engine/class_feature_data.py` and served read-only via `/api/content/spells`, `/api/content/items`, and `/api/content/class-features` (now merged with `CustomContent` rows, see above). `magic_items.json` was generated by converting 5etools' open data export (not part of this repo — it lived outside the project when generated) plus the user's own homebrew items appended on top; there is no in-repo script that regenerates it. The PDF importer cross-references both DBs by name to enrich/correct what it parses from the PDF — e.g. an inventory item matching a DB entry has its `charges`/`granted_spells`/`description` overwritten by the DB's (more reliable) values rather than trusting the PDF's printed weapon-table notes.
